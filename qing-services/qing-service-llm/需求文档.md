# 个人财务流水整合系统 - 需求文档

## 1 项目概述

### 1.1 项目目标

整合来自多个支付渠道（银行卡、支付宝、微信、云闪付、钱迹记账、京东等）的交易记录，构建统一的个人流水账本，实现：

- 任意时刻的银行卡余额查询
- 任意时刻的微信/支付宝等电子钱包余额查询
- 完整的支付记录汇总与分类
- 跨渠道交易的自动关联与去重

### 1.2 数据源清单

| 渠道   | 目录        | 文件格式             | 数据特点            |
|------|-----------|------------------|-----------------|
| 银行流水 | `银行流水记录/` | .xls, .pdf, .txt | 格式不统一，需单独解析     |
| 支付宝  | `支付宝账单/`  | .csv (GBK编码)     | 按年度分段，字段完整      |
| 微信   | `微信账单/`   | .xlsx            | 按年度分段，部分信息模糊    |
| 京东   | `京东账单/`   | .csv             | 购物记录            |
| 钱迹   | `钱迹账单/`   | .csv             | 手动记账，分类详细，有账户信息 |
| 云闪付  | 需用户提供     | 需确认              | -               |

## 2 功能需求

### 2.1 数据标准化

#### 2.1.1 统一数据结构

所有渠道的流水记录需要转换为统一的标准化格式：

```json
{
  "id": "唯一标识符",
  "transactionTime": "交易时间 (ISO 8601)",
  "channel": "渠道 (ALIPAY/WECHAT/BANK/QIANJI/JD/CLOUDPAY)",
  "account": "账户名称",
  "accountType": "账户类型 (DEBIT/CREDIT/WALLET)",
  "type": "交易类型 (INCOME/EXPENSE/TRANSFER)",
  "amount": "金额 (正数)",
  "balance": "交易后余额",
  "counterparty": "交易对手",
  "merchant": "商户/商品描述",
  "category": "消费分类",
  "subCategory": "二级分类",
  "status": "状态 (SUCCESS/PENDING/FAILED)",
  "fee": "手续费",
  "originalId": "原始记录ID",
  "sourceFile": "来源文件",
  "remark": "备注",
  "tags": [
    "标签"
  ],
  "linkedId": "关联交易ID",
  "reconciliationStatus": "对账状态 (PENDING/MATCHED/MANUAL)"
}
```

#### 2.1.2 账户类型定义

| 类型      | 说明   | 示例              |
|---------|------|-----------------|
| DEBIT   | 储蓄卡  | 建设银行、交通银行       |
| CREDIT  | 信用卡  | 中信银行信用卡、交通银行信用卡 |
| WALLET  | 电子钱包 | 微信零钱、支付宝余额      |
| VIRTUAL | 虚拟账户 | 京东小金库等          |

### 2.2 流水合并

#### 2.2.1 跨渠道去重

对于同一笔交易在不同渠道都有记录的情况（如银行卡绑定微信扣款），需要：

- 自动识别并标记重复交易
- 保留信息最完整的记录
- 记录原始关联关系

#### 2.2.2 转账识别

识别并关联以下类型的转账：

- 银行卡 ↔ 支付宝
- 银行卡 ↔ 微信
- 支付宝 ↔ 微信
- 信用卡 ↔ 储蓄卡（还款）

### 2.3 余额查询

#### 2.3.1 银行卡余额

- 支持查询任意历史时刻的银行卡余额
- 基于银行导出的期末余额进行倒推
- 支持多卡余额汇总

#### 2.3.2 电子钱包余额

- 微信零钱余额
- 支付宝余额
- 云闪付余额
- 需要期初余额作为基准（用户手动提供或从第一笔交易推算）

### 2.4 交易完善

#### 2.4.1 信息补全

针对微信支付等交易信息模糊的记录：

- 尝试通过时间+金额与钱迹账单匹配
- 尝试通过时间+金额与银行流水匹配
- 标记无法匹配的"可疑交易"供用户确认

#### 2.4.2 消费分类

- 沿用钱迹账单的分类信息
- 对无分类的记录，根据商户/描述进行智能分类
- 支持用户手动校正分类

### 2.5 增量更新

#### 2.5.1 数据增量

支持增量导入新增的账单文件：

- 自动识别已处理的文件
- 仅处理新增或修改的文件
- 记录增量导入的时间戳

#### 2.5.2 增量查询

支持仅查询特定时间段内的交易：

- 按日期范围查询
- 按渠道查询
- 按账户查询

## 4 核心算法

### 4.1 跨渠道交易匹配

```
匹配规则（优先级）:
1. 精确匹配: 时间差<1分钟 + 金额完全相同 + 渠道不同
2. 近似匹配: 时间差<1小时 + 金额相同 + 账户关联
3. 模糊匹配: 时间差<24小时 + 金额相近 + 商户描述相似
```

### 4.2 余额推算

```
对于账户A在时间T的余额:
1. 找到账户A在T之后最近的已知余额B
2. 从B向前倒推，累加T~B之间的所有交易
3. 收入(+) 支出(-) 转账(视方向而定)
4. 结果即为T时刻的估算余额
```

### 4.3 去重策略

```
同一笔交易在多渠道出现时:
1. 优先保留: 银行流水 > 钱迹 > 支付宝 > 微信 > 云闪付
2. 银行流水信息最权威
3. 钱迹有手动分类信息
4. 电子钱包记录可能最详细
```

**文件命名**: `支付宝交易明细(YYYYMMDD-YYYYMMDD).csv`

**字段** (原始CSV):

```
交易时间, 交易对方, 商品说明, 状态, 金额, 收/支, 备注...
```

**特点**:

- GBK编码，需转码处理
- 按年度分段
- 2015-2026年数据完整

### 5.3 微信

**文件命名**: `微信支付账单流水文件(YYYYMMDD-YYYYMMDD).xlsx`

**字段**:

```
交易时间, 交易类型, 交易对方, 商品, 金额(元), 收/支, 状态, 备注...
```

**问题**:

- 部分记录"交易对方"和"商品"为空
- 2015-2026年数据完整

### 5.4 钱迹

**文件**: `QianJi_日常账本_YYYY-MM-DD_HHMMSS.csv`

**字段**:

```
ID, 时间, 分类, 二级分类, 类型, 金额, 币种, 账户1, 账户2, 备注...
```

**优点**:

- 有详细分类
- 记录了账户变动
- 手动记账，信息准确

### 5.5 京东

**文件**: `京东交易流水(申请时间...).csv`

**字段**:

```
下单时间, 商品名称, 订单金额, 支付方式, 订单状态...
```

**特点**:

- 仅购物记录
- 可作为消费分类的补充

## 7 技术约束

### 7.1 编码处理

- 支付宝CSV: GBK → UTF-8
- Windows文件名: GBK/GB2312
- 统一内部存储: UTF-8

### 7.2 文件格式支持

- CSV: 优先解析
- XLS/XLSX: 需Apache POI或类似库
- PDF: 建议转为图片或文本提取

### 7.3 性能考虑

- 首次导入: 预计10000+条记录，需优化加载速度
- 增量导入: 预计单次100-500条
- 余额查询: 需建立索引
